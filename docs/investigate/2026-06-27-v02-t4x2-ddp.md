# v0.2 Training Issues — Device Placement, DDP, and TPU Idle Timeout

## Context

`v0.2-llama-qlora` trains Llama-3.2-3B with LoRA on Kaggle T4×2 (2 × 16 GB = 32 GB).
Multiple device-placement bugs were encountered and patched incrementally, each fix hiding
the real problem. This document traces the full history and records the correct DDP solution.

- [x] Trace each incremental fix and its unintended side effect
- [x] Determine why v24 ran 11+ hours on a single T4
- [x] Establish why true DDP is impossible in a Kaggle notebook
- [x] Confirm DataParallel fallback is also broken with TRL's loss
- [x] Determine correct path forward (TPU v5e-8)

## Findings

### Bug 1 — `device_map='auto'` causes cross-device tensor error

**Original cell 7:**
```python
model = AutoModelForCausalLM.from_pretrained(MODEL_PATH, dtype=torch.float16, device_map='auto')
```

`device_map='auto'` performs **model parallelism** — it splits the model's layers across
both T4s to distribute weight memory. Llama-3.2-3B (6 GB) fit on one T4, so this was
unnecessary, but `'auto'` doesn't know that.

SFTTrainer is not designed for a split model. During the backward pass,
`_chunked_cross_entropy_loss` computed indices on `cuda:1` while the tensors it indexed
were on `cuda:0`, raising:

```
RuntimeError: indices should be either on cpu or on the same device as the indexed tensor (cuda:1)
```

### Fix 1 — `device_map='cuda:0'` (incomplete)

Pinning the entire model to `cuda:0` eliminated the layer-split. But Accelerate/SFTTrainer
still **detected two visible GPUs** and wrapped the model in `DataParallel`, which copied
it to `cuda:1` during training. The cross-device error returned via a different path.

### Fix 2 — `CUDA_VISIBLE_DEVICES=0` before `import torch` (working but wasteful)

```python
import os
os.environ["CUDA_VISIBLE_DEVICES"] = "0"
import torch  # torch now sees only one GPU
```

Setting `CUDA_VISIBLE_DEVICES` before PyTorch initialises makes the second T4 completely
invisible. SFTTrainer cannot launch DataParallel on a GPU it cannot see. The crash was
eliminated.

**Unintended consequence:** Only 16 GB of the available 32 GB VRAM is used.
With `BATCH_SIZE=1` and `GRAD_ACCUM=16`, a single-fold run took **11+ hours** and hit
the Kaggle 12-hour committed-run limit before completing.

### Why the incremental fixes were wrong

Both fixes addressed the *symptom* (multi-GPU tensor movement) rather than the *cause*:
SFTTrainer/Accelerate expect **DDP** (one full model copy per GPU, data sharded), but
`device_map` was triggering **model parallelism** (one model split across GPUs, data
replicated). These are mutually exclusive strategies and Trainer only supports DDP.

### Attempted fix — DDP via Accelerate (no `device_map` at load time) — v25

Removing both `CUDA_VISIBLE_DEVICES=0` and `device_map='cuda:0'` was the theoretically
correct DDP approach. v25 was pushed with these changes. It ran but errored in cell 8
with the same cross-device tensor error:

```
RuntimeError: Caught RuntimeError in replica 1 on device 1.
Expected all tensors to be on the same device, but got index is on cuda:1,
different from other tensors on cuda:0
```

### Why true DDP is impossible in a Kaggle notebook

**True DDP** (`DistributedDataParallel`) requires each GPU to be owned by a **separate
process** launched via `torchrun` or `accelerate launch`. Each process runs an independent
copy of the training script and communicates only via all-reduce gradient sync.

Kaggle notebooks run as a **single Python process** inside Jupyter/papermill. There is no
way to fork per-GPU processes from a notebook cell. Accelerate therefore falls back to
`nn.DataParallel` — a single-process, single-thread approach where one process moves
tensors between GPUs during forward/backward passes.

TRL's `_chunked_cross_entropy_loss` is incompatible with `DataParallel` because it
computes indices and label tensors on separate GPU replicas without the coordinating logic
DataParallel requires. The result is always the same cross-device tensor error, regardless
of how the model is loaded.

**Summary:** T4×2 with SFTTrainer in a Kaggle notebook cannot use true DDP and cannot use
DataParallel with TRL's loss. The only working configurations are:

| Approach | Works? | Notes |
|---|---|---|
| Single T4 (`CUDA_VISIBLE_DEVICES=0`) | ✓ | Slow; only 16 GB VRAM |
| DataParallel (Accelerate default) | ✗ | TRL loss incompatible |
| True DDP (`torchrun`) | ✗ | Requires multi-process launch; impossible in notebook |
| TPU v5e-8 | ✓ | XLA handles multi-chip natively; no DataParallel fallback |

### Resolution — move to TPU v5e-8

TPU v5e-8 (8 chips × 16 GB = 128 GB) avoids this entire class of problem. XLA compiles
the computation graph across all chips without any DataParallel/DDP distinction — the
runtime handles distribution transparently. SFTTrainer + Accelerate on TPU uses a
fundamentally different execution model that does not have the single-process DataParallel
limitation.

Both v0.2 (Llama-3.2-3B) and v0.3 (Gemma-2-9b) can run on TPU v5e-8 with the same
adaptations: `bf16=True`, `fp16=False`, `attn_implementation='eager'`,
`dataloader_drop_last=True`, no `device_map`.

## 2. v29 — TPU idle timeout from single-threaded pre-tokenization

### Context

v29 was the first successful TPU v5e-8 run (TPU detected, model loaded, LoRA attached).
Kaggle cancelled it with exit code 137 and the message:

```
TPU stopped after being idle for 2h0m41s > 2h0m0s
(please only use TPU VMs if you use the TPU)
```

### Investigation Checklist

- [x] Confirm cancellation was Kaggle-side (not user-initiated)
- [x] Identify what was running during the 2-hour idle window
- [x] Quantify tokenization cost for 91,962 examples
- [x] Implement and push fix (v30/v32)
- [x] Confirm v32 still hits 2h idle (`num_proc=4` ineffective)
- [x] Try deferred `xm.xla_device()` + 10K samples (v34/v35)
- [x] Confirm v35 still hits 2h idle → root cause is `import torch_xla`
- [x] Decision: abandon v0.2

### Findings

Kaggle allocates the TPU chip as soon as the kernel starts, but the TPU sits idle
while any CPU-bound setup runs. If the TPU is unused for 2 hours, Kaggle kills the job.

The bottleneck: `SFTTrainer` tokenizes the dataset internally in a single-threaded loop
before the first training step. With 91,962 swap-augmented examples and full chat
template formatting, this takes ~2 hours on a single CPU core — exactly matching the
idle timeout.

Estimated tokenization time:
```
91,962 examples × ~0.08s per example (chat template + tokenize) ≈ 7,357s ≈ 2h
```

The TPU was allocated the entire time but never received a single XLA op.

Kaggle's kernel log for cancelled TPU runs is truncated — only the first ~122 seconds
of stdout/stderr are captured, making it appear the run ended very early. The actual
wall-clock time was ~2h (confirmed by the idle timeout timestamp range in the error).

### Actions Taken

**v30 fix — pre-tokenize in cell-6 with `num_proc=4`:**

Moved tokenization out of SFTTrainer and into cell-6 using `Dataset.map()` with
parallel workers. The TPU is not allocated until cell-8; by that point, tokenization
is complete and training starts immediately.

```python
# cell-6 (v30)
train_ds = raw_ds.map(
    tokenize_fn,
    batched=True,
    batch_size=1000,
    num_proc=4,
    remove_columns=['text'],
    desc='Tokenizing',
)
```

Expected tokenization time with `num_proc=4`: ~30 min (4× speedup).
TPU idle window eliminated — training starts within seconds of cell-8 executing.

SFTConfig updated with `dataset_text_field=''` to signal SFTTrainer the dataset
is already tokenized and skip its internal tokenization pass.

### Actions Taken

**v30** — pre-tokenize in cell-6 with `num_proc=4` (~30 min for 91k examples).
`dataset_text_field=''` added to SFTConfig to skip SFTTrainer's internal pass.

**v31/v32** — added `MAX_TRAIN_SAMPLES = 50_000` cap in cell-3 (pipeline validation only).
Subsampled before tokenization → ~10 min tokenization, faster training.
v32 pushed (scriptVersionId=330909233).

**v32 outcome — 2h idle timeout again:**
```
TPU stopped after being idle for 2h0m46.502257108s > 2h0m0s
```
`num_proc=4` did not help. Hypothesis: Kaggle's papermill/notebook environment restricts
`multiprocessing.fork`, so `Dataset.map(num_proc=4)` may silently fall back to single-threaded
execution, giving no speedup over v29.

**v34** — removed `xm.xla_device()` call from cell-2 to defer TPU allocation to cell-8
(the training cell). Reduced `MAX_TRAIN_SAMPLES` to 10,000.

Cell-2 changed to:
```python
try:
    import torch_xla.core.xla_model as xm  # noqa: F401
    DEVICE_TYPE = 'tpu'
    print('torch_xla available — TPU connection deferred to training cell')
except ImportError:
    DEVICE_TYPE = 'cuda' if torch.cuda.is_available() else 'cpu'
```

**v35 outcome — 2h idle timeout again (v34 kernel):**
```
TPU stopped after being idle for 2h0m55.328193356s > 2h0m0s
```
Even with 10K samples and deferred `xm.xla_device()`, the idle timeout recurs at exactly the
same 2h mark.

### Findings (updated)

The root trigger is almost certainly `import torch_xla` itself — not the `xm.xla_device()` call
and not the tokenization volume. Kaggle appears to allocate the TPU VM and start the idle timer
the moment `import torch_xla` executes (cell-2), before any user-visible XLA ops. The 2h window
is exhausted by:

1. Imports and setup (cell-2 to cell-6): ~1–5 min
2. Pre-tokenization even with 10K samples: ~5 min
3. Llama-3.2-3B model load (cell-7): ~40 s
4. **XLA graph compilation** (first call to `trainer.train()` in cell-8): this is the likely
   remaining cost. XLA traces and compiles the full computation graph on first use. For a 3B
   model with LoRA, this can take 30–90 min — and if Kaggle's idle detector does not count
   XLA compilation as "active TPU use," the timer is still running.

The only remaining fix would be to move `import torch_xla` into cell-8, immediately before
`trainer.train()`. Even this may not work if XLA compilation itself is not counted as activity.

### Resolution

- **Status:** unresolved → **v0.2 abandoned**
- v29: cancelled after 2h TPU idle; single-threaded tokenization of 91K examples.
- v30/v32: `num_proc=4` + 50K sample cap; `num_proc=4` likely fell back to single-threaded in Kaggle; still hit 2h idle.
- v34/v35: deferred `xm.xla_device()` + 10K samples; still hit 2h idle — `import torch_xla` is the likely allocator.
- DGX Spark GB10: estimated ~100h to complete; also not viable.
- **Decision:** abandon v0.2. Skip to v0.4 (reward model zero-shot on T4 — no training required).
