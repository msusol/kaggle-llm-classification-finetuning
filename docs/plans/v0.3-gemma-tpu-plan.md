# v0.3 — Gemma-2-9b TPU v5e-8 Plan

## Goal

Validate the full SFT pipeline with `google/gemma-2-9b-it` on Kaggle TPU v5e-8.
This is the step between v0.2 (Llama-3.2-3B, T4) and the DGX Spark distillation runs.
Single-fold first; expand to all 5 folds once pipeline is confirmed.

## Why Gemma-2-9b on TPU v5e-8

- **Model:** `gemma-2-9b-it` is the proven competition backbone (used by top teams); Llama-3.2-3B is a placeholder.
- **Hardware:** TPU v5e-8 = 8 chips × 16 GB HBM = 128 GB total. Gemma-2-9b in bf16 (~18 GB) fits on a single chip with room for activations; SFTTrainer distributes data across all 8 chips automatically via Accelerate.
- **No quantization:** 18 GB model across 128 GB total means pure bf16 — no bitsandbytes, no adapter dtype issues.
- **Expected improvement:** ~0.89–0.92 OOF log loss vs ~0.95+ for 3B Llama.

## Key differences from v0.2

| Aspect | v0.2 (T4) | v0.3 (TPU) |
|---|---|---|
| Model | Llama-3.2-3B | gemma-2-9b-it |
| dtype | fp16 | bf16 |
| Quantization | none (3B fits) | none (9B fits on 128 GB) |
| Device setup | `CUDA_VISIBLE_DEVICES=0` | No CUDA pinning; Accelerate handles XLA |
| `device_map` | `cuda:0` | Not set (XLA placement) |
| `fp16` in SFTConfig | True | False |
| `bf16` in SFTConfig | False | True |
| Batch per device | 1 | 2 (× 8 chips = 16 effective) |
| Folds | 1 (validate) | 1 → then all 5 |

## TPU workflow (Kaggle UI required)

`machine_shape` in kernel metadata is **read-only** on push — Kaggle ignores it and
assigns the default accelerator (P100/None). Always follow this pattern:

1. `zsh scripts/push_notebook.sh v0.3-gemma-tpu`
2. **Immediately stop** the auto-run in the Kaggle UI.
3. Right panel → **Accelerator → TPU v5e-8**
4. **Save Version → Save & Run All**

Do not let the auto-run complete — it will run on CPU or wrong accelerator.

## Model source

Gemma-2-9b-it must be attached via the Kaggle UI first (license acceptance):

1. Open notebook editor → **+ Add Input → Models**
2. Search `google/gemma-2` → select `gemma-2-9b-it`
3. After first-time attachment, add to `model_sources` in metadata:
   `"google/gemma-2/transformers/gemma-2-9b-it/1"` (verify exact path from `/kaggle/input/`)

## Pre-tokenization (required for TPU)

SFTTrainer's internal tokenization is single-threaded and takes ~2h for 91k examples,
triggering Kaggle's 2h TPU idle timeout (confirmed in v0.2 v29 — see
`docs/investigate/2026-06-27-v02-t4x2-ddp.md §2`).

**Fix applied in v0.3 cell-6:** pre-tokenize with `num_proc=4` before the training cell:

```python
train_ds = raw_ds.map(tokenize_fn, batched=True, batch_size=1000,
                      num_proc=4, remove_columns=['text'])
```

And in SFTConfig (cell-8): `dataset_text_field=''` to skip SFTTrainer's internal pass.

Expected tokenization time: ~30 min (4× parallel). TPU training starts immediately
after cell-8 runs — no idle window.

## Exit criteria

- Pipeline runs end-to-end on TPU v5e-8 without OOM or XLA errors.
- Fold 0 OOF log loss < 0.95 (better than v0.2 Llama-3.2-3B expectation).
- `submission.csv` generated with correct shape and valid probabilities.
- Record result in `leaderboard.md` and update `docs/investigate/`.
