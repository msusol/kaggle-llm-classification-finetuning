# v0.2 — QLoRA Plan (rung 3), reusing mineral-hr-llm on DGX Spark

Train a preference classifier with QLoRA on the DGX Spark (GB10, `sparkdb62`) by
**reusing the verified harness in [`msusol/mineral-hr-llm`](https://github.com/msusol/mineral-hr-llm)**.[cite:5]
That repo already has LoRA **and** QLoRA verified working on GB10/Blackwell, a working
`Dockerfile.gb10`, and a model-agnostic training backend.

## What mineral-hr-llm gives us[cite:5]

- `train_hr_lora.py` — LoRA/QLoRA backend (causal-LM SFT). Model-agnostic via
  `--model_name <local path>`. Verified flags:
  `--max_seq_length --per_device_train_batch_size --gradient_accumulation_steps
  --num_train_epochs --learning_rate --bf16 --use_qlora --local_files_only`.
- `Dockerfile.gb10` — NVIDIA PyTorch 25.12 base, native Blackwell/CUDA 13.x.
- `scripts/run_train.sh` — wraps Docker flags/mounts; `scripts/serve_vllm.sh` for
  offline vLLM inference.
- JSONL training format: `{ "id", "instruction", "context", "output" }`.

### Container of record (GB10)
Use the nemotron project's **`Dockerfile.vllm-gb10`** as the most advanced GB10/Blackwell
image — NVIDIA PyTorch **26.04** (CUDA 13.2.1, PyTorch 2.12), `linux/arm64`,
Blackwell `sm_120`, vLLM 0.8.x+ — rather than mineral-hr-llm's older 25.12/26.02 base.
Build on the Spark: `docker build --platform linux/arm64 -f Dockerfile.vllm-gb10 ...`
(never import an x86_64 image). It serves an OpenAI-compatible vLLM API, which we reuse
for batched offline inference; align the training image to the same 26.04 base.

### GB10 gotchas to carry over[cite:5]
- **QLoRA works** but `bitsandbytes` must be **built from source** for the container's
  CUDA (13.2.x) — pip wheels lack GB10 support; see the mineral repo's `plans/qlora.md`.
- **Do NOT** pass `--gradient_checkpointing` on GB10 (library incompat).
- Prefer `--bf16`; keep seq-len/batch conservative at first.
- Default to plain LoRA; switch on `--use_qlora` only when memory demands it (i.e. if
  we move from Llama-3.1-8B to gemma-2-9b / larger).
- Note the nemotron project found GB10 needs `--privileged -e NVIDIA_VISIBLE_DEVICES=all`
  rather than `--gpus all --runtime=nvidia`; reconcile the Docker run flags on first launch.

## Model choice

The harness is wired and verified for **`meta-llama/Llama-3.1-8B-Instruct`**, so the
fastest first contender is Llama-3.1-8B (reuse as-is). The published LMSYS winners
favored **`gemma-2-9b-it`** for this exact task.[cite:3][cite:4] Path:
1. **v0.2a** — train Llama-3.1-8B with the existing harness (zero porting). Establish OOF.
2. **v0.2b** — point `--model_name` at a local `gemma-2-9b-it` (needs QLoRA + tokenizer/
   chat-template check) and compare OOF.

## Two ways to frame the 3-class task

The mineral harness is **causal-LM SFT**, not a classifier. Two options:

- **(A) Label-token readout (lowest friction).** Keep causal-LM training. `output` is
  the winner label. At inference, read the logits of the label tokens
  (`A` / `B` / `tie`) at the answer position, softmax → 3 probabilities for log loss.
  Reuses `train_hr_lora.py` unchanged.
- **(B) Sequence-classification head.** Swap base for `*ForSequenceClassification`
  (`num_labels=3`) + cross-entropy. Matches LMSYS winners but requires editing the
  training backend. Do this in v0.3 only if (A) underperforms.

Start with **(A)**.

## Data conversion (Kaggle CSV → mineral JSONL)

Write `scripts/convert_to_jsonl.py` to map each train row to:

```json
{
  "id": "<id>",
  "instruction": "You are a judge. Choose which response a human preferred: A, B, or tie.",
  "context": "PROMPT:\n{prompt}\n\nRESPONSE A:\n{response_a}\n\nRESPONSE B:\n{response_b}",
  "output": "A" | "B" | "tie"
}
```

- `output` = argmax of `winner_model_a/b/tie` → `A`/`B`/`tie`.
- **A/B swap augmentation:** emit a swapped copy (responses swapped, A↔B label swapped,
  tie unchanged) to fight position bias.
- On overflow at `--max_seq_length`, keep the **tail** of each response.
- 5-fold stratified split → `data/foldK_train.jsonl` / `foldK_eval.jsonl`.

## Training command (starting point)

```bash
# on sparkdb62, inside mineral-hr-llm checkout
bash scripts/run_train.sh \
  --model_name /workspace/models/Llama-3.1-8B-Instruct \
  --train_file data/fold0_train.jsonl \
  --eval_file  data/fold0_eval.jsonl \
  --output_dir outputs/llm-pref-fold0 \
  --max_seq_length 2048 \
  --per_device_train_batch_size 1 \
  --gradient_accumulation_steps 16 \
  --num_train_epochs 1 \
  --learning_rate 2e-4 \
  --bf16 --use_qlora --local_files_only
```

(1 epoch first — watch overfit, mirrors the repo's teacher-bootstrap guidance.)[cite:5]

## Offline Kaggle submission — HF → Kaggle packaging strategy

Kaggle disables internet during scoring; all weights must arrive as pre-declared
`model_sources` or `dataset_sources` in `kernel-metadata.json`.

### Step 1 — Push adapter to Hugging Face Hub

After each training run on `sparkdb62`, push only the **adapter** (≈50 MB):

```bash
# inside the training container on sparkdb62
huggingface-cli login   # one-time; use HF token with write access
huggingface-cli upload \
  gdataranger/llm-pref-adapter-v0.2a \
  outputs/llm-pref-fold0/adapter_config.json \
  outputs/llm-pref-fold0/adapter_model.safetensors
```

Keep the repo **private** until submission is ready. The base model stays on
HF Hub under its original name (`meta-llama/Llama-3.1-8B-Instruct` or
`google/gemma-2-9b-it`) — no need to re-upload.

### Step 2 — Mirror into Kaggle Models

Kaggle has a native HF→Kaggle model import:

1. Go to kaggle.com/models → **New Model**
2. Choose **"Import from Hugging Face"** → enter `gdataranger/llm-pref-adapter-v0.2a`
3. Kaggle pulls the adapter and hosts it at
   `gdataranger/llm-pref-adapter-v0.2a/transformers/default/1`
4. Add an entry to `kernel-metadata.json`:

```json
"model_sources": [
  "meta-llama/llama-3.1-8b-instruct/transformers/8b-instruct/1",
  "gdataranger/llm-pref-adapter-v0.2a/transformers/default/1"
]
```

The base model is already on Kaggle Models — reference it directly; no upload needed.

### Step 3 — bitsandbytes wheel

Source-built `bitsandbytes` for CUDA 13.x must be uploaded separately as a
Kaggle Dataset (it is not pip-installable offline):

```bash
kaggle datasets create \
  --dir /path/to/bitsandbytes-dist \
  --title "bitsandbytes-gb10-wheel" \
  --slug bitsandbytes-gb10-wheel
```

Add to `dataset_sources` in `kernel-metadata.json`.

### Step 4 — Inference notebook

```python
# /kaggle/input/<base-model>/   ← base weights (loaded with local_files_only=True)
# /kaggle/input/<adapter>/      ← adapter_config.json + adapter weights
# /kaggle/input/bitsandbytes-gb10-wheel/  ← install before importing bitsandbytes

import subprocess
subprocess.run(["pip", "install", "--no-index",
                "/kaggle/input/bitsandbytes-gb10-wheel/bitsandbytes-*.whl"])
```

Load base + adapter offline, **swap-TTA** (A,B)+(B,A) averaged,
batch over ~25k rows < 9 h, softmax label-tokens → `submission.csv`.

### Step 5 — Run submission checklist

Run [submission-checklist.md](submission-checklist.md) before every `kaggle kernels push`.

## Acceptance

- OOF log loss beats v0.1 baseline; target ~0.89–0.92 once on gemma-2-9b.[cite:3][cite:4]
- Inference verified < 9 h on the example test set before the full run.

## Training venue options

| Venue | Pro | Con |
|---|---|---|
| **DGX Spark (sparkdb62)** | GB10 HBM (96 GB), no time limit, full control | Manual packaging to HF/Kaggle |
| **Kaggle (GPU T4×2 or P100)** | Data already mounted, direct submission, no packaging step | 30 h/week GPU quota, slower than GB10, 9 h runtime cap |

**Recommended sequence:**
1. **First run on Kaggle** — validates the full pipeline end-to-end (data loading →
   training → `submission.csv`) within the competition environment, zero packaging friction.
   Use Llama-3.1-8B with LoRA (no QLoRA needed on T4×2 for 8B at short seq-len).
2. **Scale on DGX Spark** — once the pipeline is validated, train longer / larger
   (Gemma-2-9b, more folds, more epochs) on GB10, then package via HF→Kaggle.

For the Kaggle training run, enable GPU + internet in `kernel-metadata.json`:
the training notebook can `pip install peft trl` and pull the base model from HF
at training time (internet IS available during manual notebook runs; only disabled
at submission scoring time).
